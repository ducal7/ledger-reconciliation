# Ledger Reconciliation

Double-entry ledger reconciliation over **synthetic** financial transactions, built as a
portfolio project for data analysis / data engineering work in fintech.

All data here is generated by a seeded script in this repo. There are **no external datasets,
no employer data, and no credentials** anywhere.

---

## Why this matters (the problem)

In any payments or fintech system there are (at least) two sources of truth:

1. The **internal ledger** - the company's own record of every economic event. A correct
   ledger uses **double-entry bookkeeping**: every transaction is split into balanced
   *debit* and *credit* legs, and for each transaction `sum(debits) == sum(credits)`. This
   invariant is what lets you trust that money is never created or destroyed by accident.
2. The **external feed** - what the bank / card network / settlement processor actually did
   (the "money movement" truth).

These two never agree perfectly. **Reconciliation** is the process of matching them and
classifying every discrepancy ("break"), because each break type maps to a different real
financial risk:

| Break type | Real-world meaning | Risk |
| --- | --- | --- |
| `AMOUNT_MISMATCH` | Bank settled a different amount than the ledger recorded | Revenue leakage / overpayment |
| `STUCK_UNSETTLED` | Posted internally but the bank never settled it (pending) | Cash not actually received |
| `DUPLICATE` | The same settlement posted more than once externally | Double-counted cash |
| `EXCESS_CREDIT` | Money present externally with no matching internal txn | Unexplained / unbooked funds |

The headline number every finance/ops team wants is the **at-risk (recoverable) amount**: the
total money tied up in breaks.

---

## What this repo does

- **Generates** a balanced internal double-entry ledger plus an external settlement feed,
  with a fixed, seeded set of injected anomalies (`src/recon/data.py`).
- **Validates** typed records with `pydantic` and enforces the double-entry invariant both
  per transaction and as a global account roll-up (`src/recon/models.py`).
- **Reconciles** internal vs external using **DuckDB** set-based SQL (a `FULL OUTER JOIN` of
  the internal roll-up against the aggregated feed) and classifies each break
  (`src/recon/reconcile.py`).
- **Emits an audit report** in markdown + CSV with the at-risk headline (`reports/`).

### Architecture

```
seeded generator ─► ledger.csv ─┐
(recon.data)        external.csv ┤─► pydantic validation ─► double-entry checks
                                 └─► DuckDB FULL OUTER JOIN ─► classified breaks ─► audit report
```

## How to run

Requires Python 3.11.

```bash
python -m venv .venv
source .venv/bin/activate
pip install -e ".[dev]"

# 1. Generate synthetic data (deterministic, seeded)
make data            # or: python -m recon.data

# 2. Reconcile and write the audit report
make reconcile       # or: python -m recon.reconcile

# Or do everything (generate, reconcile, lint, test)
make all

# A unified CLI is also installed:
recon all
```

Lint and test:

```bash
make lint            # ruff check
make test            # pytest
```

## Results (sample audit report)

Running the default seed (`20260626`) over 80 clean transactions plus 18 injected anomalies
produces the committed report at [`reports/audit_report.md`](reports/audit_report.md):

| Break type | Count | Amount (USD) |
| --- | ---: | ---: |
| AMOUNT_MISMATCH | 6 | 115.08 |
| STUCK_UNSETTLED | 5 | 14,704.38 |
| DUPLICATE | 3 | 6,940.78 |
| EXCESS_CREDIT | 4 | 4,527.27 |
| **TOTAL** | **18** | **26,287.51** |

- **At-risk / recoverable amount: $26,287.51**
- Transactions reconciled clean: **80** ($217,348.82)

The break counts are deterministic for a fixed seed and are asserted in the end-to-end test.

## Tests

`pytest` covers:

- **Generator determinism** - same seed gives byte-identical output; different seeds diverge.
- **Double-entry invariant checker** - a deliberately unbalanced transaction is rejected, and
  a global roll-up imbalance is caught.
- **Each break type** - crafted fixtures confirm `AMOUNT_MISMATCH`, `STUCK_UNSETTLED`,
  `DUPLICATE`, and `EXCESS_CREDIT` are each classified correctly.
- **End-to-end** - a fixed seed yields the expected break counts and a positive at-risk amount.

## Production analog

This project uses DuckDB so the whole reconciliation runs locally on a single file with no
infrastructure. In a real fintech deployment the same set-based logic - the `FULL OUTER JOIN`
of an internal ledger roll-up against an external settlement feed, with break classification -
would run as **BigQuery** SQL over warehoused production ledgers (partitioned by date,
scheduled, and alerting on the at-risk amount). The Python/pydantic layer maps to the
ingestion and contract-validation stage upstream of the warehouse.
